1. Prior status
---------------

Before TOVR-11:

- Vector-specific merge and hygiene tests already passed locally, but the
  application-layer vector path still had two production-safety gaps:
  degraded/slow vector outcomes could still flow into merged results and the
  normal per-query cache, and the backoff state only lived in-process instead
  of across requests.
- `VC-PURE` and `VC-UNIT` were both blocked by one unrelated observability
  contract failure in `AssistantReportController.php` due raw `@message`
  placeholders.
- TOVR-10 had already shown the broader Pinecone picture remained mixed:
  hosted vector structure/queryability existed, but timeout governance was
  still unproven and local direct vector queries still failed.

2. Post-change status
---------------------

Overall readiness verdict

- Retrieval-layer hardening: improved.
  - `FaqIndex` and `ResourceFinder` now compute an explicit lexical-vs-vector
    decision map before any vector call.
  - Degraded and backoff vector outcomes no longer merge into results and no
    longer populate the normal 5-minute query cache.
  - Cross-request backoff is now cache-backed and keyed separately for FAQ and
    resource retrieval.
  - Slow vector responses beyond `MAX_VECTOR_MS` are treated as degraded rather
    than silently accepted.
- Pinecone query-time hardening: improved.
  - Query-only Pinecone transport timeouts are now configurable and wired
    through a timeout-aware client used only in `Pinecone::query()`.
- Validation unblocker: fixed.
  - The raw `@message` placeholders in `AssistantReportController` were
    replaced with metadata-only exception rendering, and both `VC-PURE` and
    `VC-UNIT` now pass.
- Environment-wide enablement readiness: still `no`.
  - Local direct vector queries still fail with the
    `ProviderProxy` unregistered-caller identity error after cache rebuild.
  - The embeddings request path still depends on the shared
    `ai.settings.request_timeout` setting rather than the new query-only
    Pinecone timeout controls.
  - Hosted post-change runtime probes were not rerun in this prompt, so the
    final enablement claim cannot move past `Partially Fixed`.

4. Trigger-path map
-------------------

FAQ and resource retrieval now follow the same bounded decision structure:

1. Run lexical Search API retrieval first.
2. Build a vector decision map with:
   - `enabled`
   - `should_attempt`
   - `reason`
   - `lexical_count`
   - `best_lexical_score`
3. Skip vector search with a cacheable healthy outcome when:
   - `reason=disabled`
   - `reason=sufficient_lexical`
4. Attempt vector search when:
   - `reason=sparse_lexical`
   - `reason=low_quality_lexical`
5. Before querying Pinecone-backed Search API, check cache-backed backoff:
   - active backoff => `status=backoff`, lexical-only, no normal query-cache
     write
6. Treat these outcomes as degraded:
   - vector index unconfigured
   - vector index unavailable
   - metric validation failure
   - vector exception / transport failure
   - `elapsed_ms > MAX_VECTOR_MS`
7. On degraded outcome:
   - activate backoff
   - preserve lexical-only results
   - do not merge vector items
   - do not write the normal query cache
8. On healthy or healthy-empty outcome:
   - merge with lexical-priority rules
   - preserve minimum lexical presence
   - allow normal query-cache write
