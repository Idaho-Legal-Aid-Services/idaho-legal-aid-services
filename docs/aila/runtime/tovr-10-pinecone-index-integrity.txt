TOVR-10 Pinecone Index Integrity and Refresh Readiness
======================================================

Prompt ID: TOVR-10
Roadmap item: Tooling + Pinecone remediation -> Pinecone integrity #2
Verification level tag: repo-inspected / runtime-verified / hosted-read-only-verified / timed-query-probed / tests-rerun
Created: 2026-03-17
Last updated: 2026-03-17

1. Prior status
---------------

Before TOVR-10, TOVR-09 had already established the Pinecone inventory baseline:

- `local`, `dev`, `test`, and `live` all reported Pinecone key presence in
  effective runtime, and `vector_search.enabled=false` everywhere.
- Hosted `dev` / `test` / `live` had enabled lexical + vector Search API
  indexes with successful sample vector queries.
- `local` remained drifted with missing lexical indexes, pending
  `ilas_site_assistant` updates `10008` / `10009` / `10010`, and blocked vector
  queries caused by the Gemini provider identity path.
- Repo config and service code already proved the intended structural contract:
  `pinecone_vector` on cosine similarity with 3072 dimensions, managed by
  `VectorIndexHygieneService` and resolved through
  `RetrievalConfigurationService`.

What TOVR-09 did not yet prove:

- whether the current index counts could still be trusted as fresh
- whether hygiene snapshots matched live Search API status in each environment
- whether current refresh procedures were sufficient for current steady-state
- whether direct query timings and degraded Pinecone behavior were acceptable
- whether the overall index layer was ready for enablement

2. Post-investigation status
----------------------------

Overall readiness verdict

- Hosted structural state is mixed:
  - `live`: structurally valid, populated, queryable, and currently
    non-overdue.
  - `test`: structurally valid, populated, queryable, and currently due but
    not overdue.
  - `dev`: structurally valid, populated, queryable, but its current hygiene
    snapshot is now overdue and therefore operationally degraded.
- `local`: not structurally governable from active config and not operationally
  refreshable because `retrieval` resolves to `null`, lexical indexes are
  missing, pending updates remain, direct vector queries fail, and the stored
  hygiene snapshot is overdue.
- Timeout and degraded-response handling: not sufficient for enablement proof.
  Repo code proves graceful exception fallback and backoff, but there is still
  no repo-owned preemptive Pinecone transport timeout to validate against.
- Index-layer enablement readiness: `no`.

Interpretation

- Hosted timings prove the query path is live and responsive enough for further
  evaluation, but they do not prove acceptable timeout governance because the
  repo still relies on provider-default Pinecone HTTP timeouts.
- `FaqIndex` and `ResourceFinder` do enforce post-failure backoff plus graceful
  fallback, but the timeout budget is request-duration observation, not a
  configured Pinecone transport limit.

  rebuild proof.

  all non-live environments.

  read-only there.

  directly inspected.

  a clean Pinecone transport timeout control to exercise.
