## Residual risks / unknowns

- The new-session flood budget is keyed by resolved client IP, so large shared
  NAT/proxy populations may still consume the anonymous bootstrap budget faster
  than a true per-browser identity model would allow.
- The rolling snapshot is intentionally coarse and state-backed: it captures
  aggregate new-session / denied counts and active thresholds, not per-IP or
  per-request forensic detail.
- Same-session bootstrap reuse remains intentionally allowed without rate
  limiting so long-lived anonymous browser sessions can recover CSRF continuity
  without churning new sessions.

## Rollback notes

- Code rollback: revert the service/controller/config/test/doc changes listed
  above.
- No schema migration or data backfill was introduced, so no database rollback
  is required.
- If the new bootstrap limits prove too tight in production-like traffic,
  thresholds can be relaxed through the exported `session_bootstrap` config
  block without changing the route contract.
