# TOVR-15 Privacy and Analytics Boundary Remediation (2026-03-18T00:48:13Z)

- Prompt ID: `TOVR-15`
- Verification level tag: `repo + local runtime post-change + hosted runtime read-only pre-deploy + targeted tests`

## Prior status

- Assistant telemetry boundaries were already mostly minimized on the
  server-controlled side:
  - Drupal analytics/conversation storage had already been reduced to hashes,
    IDs, buckets, and signatures.
  - Sentry payload shaping already scrubbed PII and promoted only approved tags.
  - Langfuse payload shaping already kept tracing to minimized, metadata-first
    summaries.
- The remaining boundary problem was GA on assistant surfaces:
  - `web/modules/custom/ilas_site_assistant/js/assistant-widget.js` still pushed
    assistant-originated events into `window.dataLayer`.
  - Hosted `live` `/assistant` still rendered `googletagmanager`,
    `window.dataLayer`, and `gtag()`.
- Safe baseline proof before the remediation:
  - Local `/assistant` already showed Sentry/browser observability markers and
    no GA markers.
  - Hosted `dev` and `test` `/assistant` also showed no GA markers.
  - Hosted `live` `/assistant` still showed GA markers and therefore exceeded
    the desired assistant-only privacy boundary.

## Post-change status

- Assistant widget tracking no longer writes assistant-originated telemetry to
  `window.dataLayer`.
- Dedicated `/assistant` page rendering now suppresses GA bootstrap on route
  `ilas_site_assistant.page`, even when `google_tag_id` is present for sitewide
  live analytics.
- `ilas:runtime-truth` now distinguishes:
  - sitewide `google_tag_id_present`
  - `browser_expected.google_analytics.tag_present`
  - assistant-page suppression expectations:
    - `assistant_page_suppressed=true`
    - `assistant_page_loader_expected=false`
    - `assistant_page_data_layer_expected=false`
- Local post-change proof is clean:
  - `/assistant` rendered only Sentry/browser observability markers.
  - No `googletagmanager`, `dataLayer`, or `gtag()` markers appeared locally.
- Hosted post-change status is intentionally split:
  - `dev` and `test` already behave acceptably for `/assistant` and still show
    no assistant-page GA markers.
  - `live` remains `repo remediated / deployment pending`; the currently
    deployed `/assistant` page still shows old GA bootstrap markers because the
    new repo code has not been deployed there yet.
  - A non-assistant hosted `live` page (`/`) still shows GA markers, which is
    acceptable under the TOVR-15 policy because the deny rule is specific to
    assistant-originated telemetry and the dedicated assistant route.

## Payload-boundary matrix

| Telemetry path | Destination | Policy | Data allowed to leave Drupal-controlled storage | Data denied from leaving Drupal-controlled storage | Current proof |
|---|---|---|---|---|---|
| `/assistant/api/track` -> `AnalyticsLogger` | Drupal DB | `allow` | normalized event name, route/path context, conversation/session IDs, coarse metadata | raw user message text, legal-aid-sensitive free text, richer analytics payloads | code inspection + existing analytics/kernel contracts |
| Conversation logging | Drupal DB | `allow` | hashed/minimized conversation metadata, deterministic IDs/signatures | raw assistant/user content export to third parties | code inspection + existing conversation/kernel contracts |
| Browser custom events (`ilas:assistant:*`) | Browser only / internal listeners | `allow with constraint` | transient minimized action/error metadata used by internal observability hooks | forwarding raw content to GA or other third parties | widget/observability code inspection |
| Browser Sentry | Sentry | `allow with constraint` | approved tags, scrubbed extra data, browser health/error metadata | user content, email, IP, unsanitized extras | existing Sentry contract tests + browser/runtime proof |
| PHP Sentry | Sentry | `allow with constraint` | approved tags, scrubbed exception/log metadata | raw legal-aid-sensitive text, direct PII | existing Sentry contract tests |
| Langfuse | Langfuse | `allow with constraint` | hashed/minimized trace summaries, counts, flags, degraded reasons, retrieval metadata | raw user message text, rich conversation bodies, expanded retention payloads | existing Langfuse contract tests |
| Assistant widget -> `window.dataLayer` / GA4 | Google Analytics | `deny` | none | all assistant-originated interaction telemetry | widget code change + JS/unit contracts |
| `/assistant` HTML GA bootstrap | Google Analytics | `deny` | none on the assistant route | `googletagmanager`, `dataLayer`, `gtag()` on `/assistant` | theme route suppression + local runtime proof |
| Non-assistant site pages with sitewide GA | Google Analytics | `allow outside assistant scope` | ordinary sitewide GA as separately governed by site policy | assistant-originated telemetry routed through widget/page | hosted `live` `/` curl |
| New Relic browser/change tracking | New Relic | `deny / keep disabled` | none | all assistant browser RUM/change-tracking data | retired/dormant state remains unchanged |
